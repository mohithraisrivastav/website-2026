01 / Accessibility — WCAG 2.1 AA
Critical
Hamburger button is a <div>, not a <button> — no aria-label
index.html · about.html · contact.html · shop.html · all pages
The hamburger is a <div> with an onclick handler. It has no role, no aria-label, and cannot be reached or triggered by keyboard users. WCAG 2.1.1 and 4.1.2 both fail.
Current (broken)
<div class="hamburger" id="hamburgerBtn" onclick="toggleMenu()"> <span></span><span></span><span></span> </div>
Fix
<button class="hamburger" id="hamburgerBtn" onclick="toggleMenu()" aria-label="Open navigation menu" aria-expanded="false" aria-controls="mobileMenu"> <span></span><span></span><span></span> </button>
Also toggle aria-expanded and aria-label to "Close navigation menu" in toggleMenu().
Critical
Cart icon on index.html — no role, no aria-label, not a button
index.html line 469
The .cart-icon div is used only on index.html. It has no ARIA role and no accessible label. The .nav-cart anchor variant (used on all other pages) correctly has aria-label="Cart". Standardise to the accessible variant.
Current (broken)
<div class="cart-icon" onclick="toggleCart()"> <svg ...></svg> <span class="cart-count"></span> </div>
Fix — use the same pattern as other pages
<a href="#" class="nav-cart" onclick="event.preventDefault();toggleCart();" aria-label="Open cart" role="button"> <svg ...></svg> <span class="cart-count" id="cartCountGlobal"></span> </a>
Critical
Cart close button is a <div>, not a <button>
All pages — cart sidebar
<div class="close-cart" onclick="toggleCart()">Close</div> is not keyboard-accessible. Replace with a <button>.
Fix
<button class="close-cart" onclick="toggleCart()" aria-label="Close cart">Close</button>
Critical
No skip-to-main-content link
All pages
Keyboard users must tab through the entire nav on every page before reaching content. WCAG 2.4.1 requires a mechanism to skip repeated navigation blocks.
Fix — add this as the very first element inside <body>
<a href="#main-content" class="skip-link">Skip to main content</a> <!-- CSS --> .skip-link { position: absolute; top: -40px; left: 0; background: #1A1612; color: #F7F3ED; padding: 8px 16px; z-index: 99999; text-decoration: none; font-weight: 700; transition: top 0.2s; } .skip-link:focus { top: 0; }
Critical
Hero Vimeo iframe has no title attribute
index.html line 516
Screen readers announce iframes by their title. Without it, users hear "unnamed frame." WCAG 4.1.2 fails.

Since it's decorative, aria-hidden="true" and tabindex="-1" also hide it from AT entirely — even cleaner.
Critical
Footer newsletter input missing label — on most pages
about.html · contact.html · all pages except index.html
The footer email input has no associated <label>. Only index.html has the correct sr-only label. WCAG 3.3.2 fails. The placeholder alone is not sufficient.
Current (broken — about.html, contact.html)

Fix — match index.html pattern

Critical
:focus { outline: none } removes all focus indicators globally
assets/css/typo.css line 97
:focus { outline: none; } strips focus rings for older browsers and some assistive tech that don't support :focus-visible. WCAG 2.4.7 requires visible focus indicators.
Current (broken)
:focus { outline: none; } :focus-visible { outline: 2px solid #CFA246; ... }
Fix — remove the :focus rule entirely; :focus-visible is enough
/* Remove the :focus { outline: none } rule. :focus-visible already handles modern keyboard focus. */ :focus-visible { outline: 2px solid #7F5F10; /* use dark gold — better contrast on light bg */ outline-offset: 3px; border-radius: 2px; }
Also switch the outline color from #CFA246 to #7F5F10. The gold #CFA246 achieves only 2.15:1 contrast on the parchment background — WCAG fail.

Major
Most GSAP scroll animations ignore prefers-reduced-motion
index.html · about.html · all pages with GSAP
Only the background parallax text checks prefers-reduced-motion. All other scroll reveals, hero entrances, and stagger animations run regardless of user preference. WCAG 2.3.3 (AAA) and general best practice.

06 / Inconsistent Spacing
Moderate
Mixed spacing units with no clear convention
index.html inline styles · about.css · global.css
Padding and margin values use four different unit types without a clear rule for when each applies. This makes spacing feel inconsistent at different viewport sizes and makes maintenance difficult.
/* px — fixed, doesn't scale */ .testimonials { padding: 80px 5vw; } /* % — relative to parent width */ .about-manifesto { padding: 8vh 10%; } /* vw — relative to viewport width */ .pinned-section { padding: 7vh 8vw 10vh; } /* vh — relative to viewport height */ .section-padding { padding: 15vh 5vw; }
Recommended convention
/* Use clamp() for vertical rhythm — scales with viewport, has hard limits */ padding-block: clamp(48px, 8vh, 120px); /* Use vw (or %) consistently for horizontal gutters */ padding-inline: 5vw; /* Reserve px for fine details: borders, icon sizes, gaps within components */
Moderate
Excessive use of !important in responsive overrides masks base rule conflicts
index.html inline style · global.css · about.css
There are 50+ !important declarations across index.html's inline <style> block alone. Most of these exist to fight specificity conflicts caused by having inline styles, page-specific CSS, and global.css all defining the same properties. The root cause is the inline <style> block pattern — moving all page styles to their proper CSS file would eliminate most of these.
Info
index.html has 350+ lines of inline CSS in <style> tags
index.html lines 71–423
All page-specific styles for index.html live in an inline <style> block. These cannot be cached by the browser across page visits. Moving them to a proper assets/css/pages/index.css file (which already has partial files for other pages) would improve cache performance and make the HTML file dramatically more readable.
Info
rel="noopener" missing "noreferrer" on social links
index.html · all pages — social links in footer
External links use rel="noopener" which prevents the new tab from accessing the opener window. Adding noreferrer additionally prevents passing the referrer header for better privacy.
Fix
<a href="https://www.instagram.com/..." target="_blank" rel="noopener noreferrer">Instagram</a>
